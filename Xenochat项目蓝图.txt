# Xenochat 项目最终极致完美整合计划

本计划整合了原 MaiBot 生态五份深度分析报告、用户自拟七十条要点、前期详细开发路线以及多次评审意见，形成一份完整、可审计、可执行的终极方案。它完整继承了原生态所有优点，彻底清除了所有已知缺陷，并引入原创性保障、生产级安全、Metal GPU 加速及全球 17 社交平台分级覆盖，确保项目达到“极致完美”标准。

---

## 1. 项目总目标与核心红线

### 1.1 总目标
构建一个**后端核心基于 Rust、前端完全重写**的模块化、高性能、跨平台的智能聊天机器人框架 **Xenochat**，实现：
- **后端核心 Rust-first**：协议层、智能体核心、适配器全部使用 Rust 实现，确保内存安全与极致性能。
- **前端原创重写**：管理面板采用 React + TypeScript，但所有代码、界面、素材完全重新撰写，与参考项目无任何直接复制。
- **继承并超越原生态**：保留 MaiBot 所有拟人化、多模型协作、插件化等优点，消除所有安全、质量、文档缺陷。
- **全球社交平台完全覆盖**：支持 discord, googlechat, imessage, instagram, kakaotalk, line, messenger, QQ(qq), signal, skype, slack, teams, telegram, viber, wechat(微信), whatsapp, zoom 平台的全功能机器人
- **Apple Silicon 原生加速**：利用 Metal 框架和 MPS 后端实现 GPU 加速推理。
- **100% 原创**：代码、界面、素材、注释全部重新撰写，通过三层原创审计。

### 1.2 核心红线（可量化验收标准）
- **代码原创性**：代码相似度扫描（jscpd）相似度 < 10%；UI 结构差异审查通过（无相同布局）；素材指纹无匹配；注释相似度（NLP）< 20%。
- **包管理唯一性**：项目根目录下不得存在 `package-lock.json` 或 `yarn.lock`；CI 检查中若发现则失败；构建脚本中无 `npm` 命令调用；CI 中无 npm 相关步骤。
- **虚拟环境存在**：项目根目录下存在 `.Xenochat` 目录，并可通过 `source .Xenochat/bin/activate` 激活，用于运行辅助脚本。
- **合规闸门**：每个平台适配器开发前需填写合规检查表，经法务或项目负责人签字确认，并记录在 `docs/compliance/` 下。
- **依赖许可证**：使用 `cargo-deny` 检查依赖许可证。
- **密钥加密**：所有 L3 数据（密钥）在配置文件中加密存储（AES-256-GCM），运行时解密；启动时若检测到明文密钥则发出警告并退出。
- **日志脱敏**：日志输出中，所有密钥、令牌、密码均替换为 `***`；单元测试验证脱敏规则。
- **默认监听安全**：默认配置中 `host = "127.0.0.1"`；若用户修改为公网 IP，启动时打印警告并提示风险。
- **CORS 严格**：默认 `allowed_origins = []`（拒绝所有）；用户需显式配置；若配置 `["*"]` 则警告。
- **GPU 加速**：在 MacBook arm64 上运行 `xenochat-gpu` 测试，检测到 Metal 设备并成功运行至少一个模型推理，输出性能指标。
- **平台覆盖**：在对应版本（MVP/Beta/GA）中，每个平台按支持等级完成并通过验收测试（测试用例覆盖该等级所有功能）。

### 3.2 核心模块设计
- **消息模型**：统一 `Message` 结构，包含平台、会话、发送者、内容、时间戳、引用等。
- **事件总线**：基于 `tokio::sync::broadcast` 和 `mpsc`，支持顺序保证。
- **配置系统**：`config` crate + 自定义热重载，支持 TOML/YAML/环境变量。

## 5. 安全与数据治理

### 5.1 威胁建模与数据分级
- **威胁建模**：在架构阶段完成 STRIDE 分析，识别关键风险并设计缓解措施。
- **数据分级**：
  - **L0（公开）**：版本号、统计信息
  - **L1（内部）**：日志（不含用户消息）、配置（不含密钥）
  - **L2（敏感）**：用户消息内容、联系人信息
  - **L3（机密）**：API 密钥、访问令牌、加密密钥
- 各级数据采取不同保护措施：L2 及以上存储加密，L3 仅内存中解密，日志脱敏。

### 5.2 密钥管理
- **存储**：敏感密钥加密后存储在本地配置文件（使用 `aes-gcm`，密钥派生自用户提供的主密码或系统 Keychain）。
- **使用**：程序启动时解密到内存，运行期间不落盘。
- **轮换**：支持密钥热更新，旧密钥可平滑过渡。
- **macOS Keychain 集成**：可选将主密码存储在 Keychain 中，实现无感解锁。

### 5.3 服务等级目标（SLO）
- **可用性**：核心服务 99.9%（每月停机 < 43 分钟）。
- **端到端延迟**：消息从平台接收到核心处理完成，p95 < 200ms（不含 LLM 推理）。
- **消息丢失率**：< 0.1%（可接受因网络中断导致的丢失，但需有重试和告警）。
- **恢复时间**：崩溃后自动重启 < 5 秒；数据损坏可回滚至最近 checkpoint。

### 5.4 数据生命周期管理
- **导入**：用户数据导入时建立索引，支持增量更新。
- **存储**：消息数据可配置存储时间，超期自动归档或删除。
- **归档**：支持导出为 JSON/SQLite 文件，加密存储。
- **删除**：提供“可证明擦除”功能，彻底删除用户数据并记录操作。
- **审计**：所有数据操作记录日志，可追溯。

## 8. 原创性保障与命名规范

### 8.1 命名规范
- **项目名**：Xenochat（所有文档、代码中使用此名）。
- **Rust crate 前缀**：`xenochat-`。

- **配置文件**：`xenochat.toml`（主配置），`models.toml`，`platforms.toml`。
- **二进制名**：`xenochat`。

### 8.2 代码原创规则
- **禁止复制**：任何代码不得从原项目直接复制粘贴，即使一行。必须重新输入或基于文档重写。
- **禁止直译注释**：注释必须重新撰写，表达相同意思但用不同英文句式。
- **禁止相同布局**：前端组件结构、样式类名、状态管理逻辑必须与参考项目有显著差异（可通过重构实现）。
- **素材原创**：使用开源图标库（如 `lucide-react`）并重新组合；背景图、插画使用 AI 生成或自行设计。

### 8.3 原创审计执行规则
- **代码相似度**：使用 `jscpd` 对比参考项目，阈值 < 10%，若超过需人工审查并重构。
- **UI 结构差异**：使用 `puppeteer` 截图对比，要求布局结构差异显著（如组件排列、类名完全不同）。
- **素材指纹比对**：所有图片、图标生成 hash，与参考项目资源比对，无相同 hash。
- **注释原创度**：使用 NLP 工具计算注释与参考项目注释的相似度，阈值 < 20%（避免直译）。CI 中自动检查。

### 8.4 许可证合规
- **选择许可证**：采用 AGPL3.0，确保严格分发。
- **第三方依赖**：通过 `cargo-deny` 检查许可证兼容性（关联 FIX-022）
- **NapCat 限制条款**：由于我们完全独立实现，不涉及 NapCat 代码，因此不受其限制。

## 9. 验收标准（八大轴）

| 轴 | 验收标准 |
|----|----------|
| 功能全覆盖 | 所有 REQ 需求卡在对应版本中实现，并通过测试。测试用例覆盖每个需求的核心功能，通过率 100%。 |
| 优点全继承 | 优势清单 ADV 中的每一点在设计中体现，并通过代码审查确认。每个优点至少有一个对应的实现模块或机制。 |
| 缺点全消除 | 缺陷清单 FIX 中的每一条已修复，安全扫描无 P0 漏洞，P1/P2 漏洞修复率 100%。启动时自检无安全告警。 |
| 原创合规 | 相似度扫描通过（<10%），素材指纹无匹配，注释相似度 <10%，UI 截图对比差异显著。 |
| pnpm-only | 前端依赖通过 pnpm 管理，项目中无 `package-lock.json` 或 `yarn.lock`；CI 检查通过。 |
| `.Xenochat` 存在 | 根目录下存在有效的 Python 虚拟环境，并可通过 `source .Xenochat/bin/activate` 激活，用于运行辅助脚本。 |
| Metal/MPS 可用 | 在 MacBook arm64 上运行 `xenochat-gpu` 测试，检测到 Metal 设备并可执行至少一个模型推理，输出性能指标。 |
| 平台分级覆盖 | MVP：4 个 L2/L3 平台（Telegram, Discord, Slack, Teams）可用，并通过验收测试；其他平台至少 L1 导入。Beta：8-10 个 L2；GA：17 平台覆盖，其中完全 17 个 L3。 |

## 10. 发布后持续改进路线

- **每周**：运行 `cargo-audit`、`pnpm audit`，生成安全报告；检查 SLO 达成情况。
- **每月**：执行性能基准测试，与上月对比，分析退化；更新技术债清单。
- **每季度**：平台兼容性升级；小版本发布。
- **每半年**：进行一次完整的安全审计（可能聘请第三方）；更新威胁模型。
- **持续**：维护风险台账，跟踪 TODO 进度，逐步偿还技术债。
